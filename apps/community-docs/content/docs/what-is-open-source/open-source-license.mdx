### **소프트웨어 라이선스란?**

- 소프트웨어의 소스 코드를 누구나 열람, 사용 수정 재배포할 수 있도록 허용하는 법적 약속
- 저작권자가 소프트웨어의 사용, 복제, 배포, 수정 등에 대해 특정 조건을 명확히 규정.

### 🔍 주요 오픈 소스 라이선스 비교

| 라이선스       | 유형             | 상업적 사용 | 수정 및 재배포 | 특이사항                             |
| -------------- | ---------------- | ----------- | -------------- | ------------------------------------ |
| **MIT**        | 매우 높음        | 가능        | 가능           | 라이선스 및 저작권 고지 포함 필수    |
| **Apache 2.0** | 높은             | 가능        | 가능           | 특허 라이선스 포함, 명시적 고지 필요 |
| **GPL v2/v3**  | 제한적(Copyleft) | 가능        | 가능           | 동일 라이선스로 재배포 의무          |

### MIT 라이선스

- 가장 간단하고 자유로운 라이선스
  - 소프트웨어를 누구나 자유롭게 사용, 수정, 배포 가능
- 단, 소프트웨어 모든 복사본에 원본 라이선스와 저작권 고지를 포함

### Apache License 2.0

- 높은 자유도와 더불어 특허 보호까지 제공하는 라이선스
- 사용, 수정, 배포, 상업적 이용이 모두 가능하며, 저작권,라이선스 고지와 변경 사항 고지 의무가 없다.

### GPL

- CopyLeft원칙이 적용된 강력한 자유 소프트웨어 라이선스
- 소스 코드 공개 및 동일 라이선스 적용이 필수.

### ✅ MIT 라이선스 준수 예시

1. **저작권 고지 및 라이선스 전문 포함**

   - 복사하거나 수정한 코드에는 원본 저작권 고지와 MIT 라이선스 전문을 포함해야 한다.
   - 예를 들어, 해당 코드가 포함된 파일의 상단에 다음과 같은 주석을 추가할 수 있다:

   ```jsx
   /**
    * Portions of this code are derived from Figma-Context-MCP by GLips.
    * https://github.com/GLips/Figma-Context-MCP
    *
    * Licensed under the MIT License.
    */
   ```

2. **변경 사항 명시 (선택 사항)**
   - 코드를 수정한 경우, 변경된 부분을 명확히 표시하는 것이 좋다.
   - 이는 원본과의 차이를 명확히 하고, 향후 유지보수에 도움.
3. **배포 시 LICENSE 파일 포함**
   - 프로젝트를 배포할 경우, 루트 디렉토리에 `LICENSE` 파일을 포함시키고, 해당 파일에 MIT 라이선스 전문과 원본 저작권 정보를 명시해야 한다.
